# 🌍 Sistema Multi-idiomas & Otimização SSR (Arquitetura)

O **BaseSaas** foi arquitetado sob uma premissa rígida de globalização: **nenhum texto visível ao usuário deve ser chumbado (*hardcoded*) no código da aplicação**. 

Todas as mensagens, títulos, rótulos e notificações são dinamicamente controlados e traduzidos usando o ecossistema de internacionalização do **Transloco**.

---

## 📂 Organização dos Arquivos de Tradução

As traduções são mantidas em arquivos JSON limpos localizados na pasta:
`public/i18n/`

Atualmente, o projeto possui suporte nativo a três idiomas (configurados em `app.config.ts` como disponíveis):
1. **[pt.json](file:///d:/Projetos/BaseSaas/public/i18n/pt.json)**: Português (Idioma Padrão / Default)
2. **[en.json](file:///d:/Projetos/BaseSaas/public/i18n/en.json)**: Inglês (Idioma de Fallback)
3. **[es.json](file:///d:/Projetos/BaseSaas/public/i18n/es.json)**: Espanhol

## 🛠️ Como Consumir Traduções no Código?

### 1. Nos Templates HTML (Via Pipe)
O Transloco expõe o pipe `transloco` para tradução de strings estáticas:
```html
<h1 class="auth-title">{{ 'AUTH.LOGIN.TITLE' | transloco }}</h1>
```

### 2. Para Renderizar Tags HTML Traduzidas (Como `<strong>` e `<code>`)
Para chaves que contenham formatações ricas em HTML, use a diretiva `[innerHTML]` do Angular para evitar escape de caracteres indesejados:
```html
<p class="error-description" [innerHTML]="'GLOBAL.CONFIG_ERROR.DESCRIPTION' | transloco"></p>
```

### 3. No Código TypeScript (Via Injeção de Serviço)
Se precisar traduzir textos no lado lógico (como mensagens de Toasters, alertas ou Guards), injete o `TranslocoService` usando a função `inject`:
```typescript
import { inject } from '@angular/core';
import { TranslocoService } from '@jsverse/transloco';

export class MeuComponente {
  private readonly transloco = inject(TranslocoService);

  exibirAlerta() {
    const mensagem = this.transloco.translate('CHAVE.DA_MENSAGEM');
    console.log(mensagem);
  }
}
```

## ⚡ Otimização Crítica para Server Side Rendering (SSR) e Prerendering

### O Problema Comum do SSR
No desenvolvimento web tradicional com Transloco, as traduções são carregadas sob demanda fazendo uma requisição HTTP para obter o arquivo JSON (`HttpClient.get('/i18n/pt.json')`).
Contudo, durante as etapas de **build de produção, Server Side Rendering (SSR) e prerenderização estática de rotas**, o compilador executa a aplicação dentro de um ambiente Node.js. Como nenhum servidor local está ativamente no ar para responder requisições na porta HTTP padrão, o carregamento do JSON de traduções quebra devido a erros de rede.

### A Nossa Solução Inteligente
Para garantir compilações ultra-rápidas sem erros ou timeouts de rede em rotas prerenderizadas, reestruturamos o `TranslocoHttpLoader` (`src/app/transloco-loader.ts`).
Utilizando o `PLATFORM_ID` fornecido pelo Angular, identificamos se o código está rodando no navegador do usuário final ou no servidor Node.js (durante o build).

```typescript
if (isPlatformBrowser(this.platformId)) {
  // No Navegador: Faz a requisição HTTP clássica
  return this.http.get<Translation>(`/i18n/${lang}.json`);
} else {
  // No Servidor/Build: Lê diretamente os arquivos JSON no disco do seu PC!
  const fs = require('fs');
  const path = require('path');
  const filePath = path.join(process.cwd(), 'public/i18n', `${lang}.json`);
  const fileContent = fs.readFileSync(filePath, 'utf8');
  const translation = JSON.parse(fileContent);
  return of(translation); // Retorna instantaneamente como Observable!
}
```

### 🎯 Benefícios
- **Compilações Seguras**: Sem falhas de rede no prerender.
- **Zero Requisições Locais Inúteis**: O carregamento em disco no Node.js ocorre em nanossegundos.
- **Transparência Absoluta**: O restante da aplicação consome os dados do mesmo jeito, sem precisar saber se as traduções vieram de um arquivo local ou de uma URL HTTP.

## 💎 Seletor de Idiomas Premium e Topbar Global

A barra superior (**[TopbarComponent](file:///d:/Projetos/BaseSaas/src/app/shared/components/topbar/topbar.component.ts)**) foi integrada de forma global no componente raiz da aplicação (`app.html`), garantindo que o cabeçalho e a troca de idioma estejam disponíveis em todas as telas da plataforma.

### 🌟 Design e Acessibilidade (Mobile-First)
- **Menu Inteligente:** Se o usuário estiver deslogado (como nas telas de login/cadastro), a Topbar exibe apenas o logotipo e o seletor de idiomas. Se logado, exibe os detalhes do perfil (avatar e nome) e o botão de logout.
- **Áreas de Toque:** Todos os botões e áreas interativas possuem tamanho mínimo de `44px` seguindo rigidamente o padrão de acessibilidade WCAG.
- **Responsividade:** Em telas móveis (`< 768px`), textos secundários como o nome por extenso do perfil são ocultados, preservando apenas o avatar e mantendo a barra 100% livre de overflow horizontal.

### 🌐 Ícone de Globo e Alinhamento Lateral
- O seletor utiliza o componente de elite `<p-select>` do PrimeNG.
- **Ícone e Texto:** Para um visual minimalista, o botão principal substitui a bandeira do país por um **ícone de globo** esmeralda (`pi-globe`) ao lado da abreviação do idioma selecionado (`PT`, `EN`, `ES`).
- **Alinhamento à Esquerda:** O painel suspenso é renderizado localmente (`appendTo="self"`) com regras CSS de prioridade absoluta (`position: absolute; right: 0; left: auto;`). Isso força o painel a se expandir **totalmente para o lado esquerdo do botão**, mantendo-se sempre dentro do viewport e evitando qualquer deslocamento lateral de tela ao clicar.

## 💾 Persistência de Preferência (`localStorage`)

### Como Funciona a Gravação?
Sempre que o usuário seleciona um novo idioma, o `TopbarComponent` atualiza o estado ativo no Transloco e persiste a preferência no navegador do cliente:
```typescript
if (typeof localStorage !== 'undefined') {
  localStorage.setItem('lang', event.value.code);
}
```

### Como Funciona a Recuperação?
O arquivo de configuração do aplicativo (**[app.config.ts](file:///d:/Projetos/BaseSaas/src/app/app.config.ts)**) lê de forma reativa a preferência do usuário no momento do bootstrap:
```typescript
defaultLang: (typeof localStorage !== 'undefined' && localStorage.getItem('lang')) || 'en'
```
Caso nenhum idioma esteja gravado (primeiro acesso), a aplicação adota o **Inglês (`'en'`)** como idioma padrão.

## ⚡ Mecanismo Antiflicker Pré-Bootstrap (SSR Hydration)

### O Problema do Flash de Idioma
Como o BaseSaaS utiliza pré-renderização estática de rotas (SSR), os arquivos HTML das páginas de login e dashboard vêm pré-renderizados do build com os textos padrão do servidor (em **inglês**). 

Quando um usuário com idioma salvo em português (`'pt'`) atualiza a página, o navegador desenha imediatamente o HTML do servidor (em inglês) e, frações de segundo depois (assim que o Angular termina de inicializar e carregar a tradução no navegador), a tela pisca e o texto muda para o português.

### A Nossa Solução Definitiva
Criamos uma transição de carregamento fluida e sem flickering visual usando o logotipo e o spinner premium de carregamento do app:

1. **Intercepção Inline Instantânea ([index.html](file:///d:/Projetos/BaseSaas/src/index.html)):**
   Injetamos um script inline extremamente leve na tag `<head>`. Este script é avaliado pelo navegador antes de começar a pintar o HTML do corpo (`<body>`). Caso haja um idioma diferente de inglês salvo no `localStorage`, adicionamos uma classe `.lang-loading` no elemento raiz `<html>`:
   ```html
   <script>
     (function() {
       try {
         var saved = localStorage.getItem('lang');
         if (saved && saved !== 'en') {
           document.documentElement.classList.add('lang-loading');
         }
       } catch (e) {}
     })();
   </script>
   ```

2. **Bloqueio Visual Temporário ([styles.css](file:///d:/Projetos/BaseSaas/src/styles.css)):**
   Definimos regras CSS estritas aplicadas apenas enquanto a classe `.lang-loading` está ativa. A tela de carregamento premium do SaaS é forçada a se manter visível e ativa, enquanto a saída real de rotas (`.app-viewport`) fica invisível:
   ```css
   .lang-loading .app-viewport {
     opacity: 0 !important;
     pointer-events: none !important;
   }
   .lang-loading .loading-screen {
     display: flex !important;
     opacity: 1 !important;
   }
   ```

3. **Exibição Pós-Boot ([app.ts](file:///d:/Projetos/BaseSaas/src/app/app.ts)):**
   Assim que o bootstrap do Angular completa a hidratação no cliente e as traduções estão 100% prontas no navegador, removemos a classe `.lang-loading` no construtor do componente raiz (`App`), revelando instantaneamente a página já traduzida de forma limpa:
   ```typescript
   constructor() {
     if (typeof document !== 'undefined') {
       document.documentElement.classList.remove('lang-loading');
     }
   }
   ```
